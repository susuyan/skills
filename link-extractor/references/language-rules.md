# 语言规则

## 基本原则

### 1. 简洁性
- 描述 ≤120 字符
- 避免冗余词汇
- 突出核心价值

**示例**:
```
❌ 这是一个非常优秀的开源项目,它能够帮助开发者快速构建应用
✅ 快速构建应用的开源框架
```

### 2. 客观性
- 陈述事实,不加主观评价
- 避免"很棒"、"优秀"、"强大"等形容词
- 使用功能性描述

**示例**:
```
❌ 这是一个很棒的 AI 工具
✅ 提供 AI 模型和 API 集成功能
```

### 3. 准确性
- 技术术语准确
- 不夸大功能
- 区分"支持"、"内置"、"提供"等表述

**示例**:
```
❌ 内置所有功能
✅ 支持 10+ 核心功能
```

## 常用表述规范

### 工具/产品
- "提供" + 功能名称
- "支持" + 特性列表
- "基于" + 技术栈

**示例**:
```
- 提供 API 集成和模型管理
- 支持多语言和自定义插件
- 基于 React 和 TypeScript 开发
```

### 文章/博客
- "深入探讨" + 主题
- "解析" + 技术概念
- "分享" + 经验/实践

**示例**:
```
- 深入探讨微前端架构在企业级应用的实践
- 解析 React Hooks 的核心原理和最佳实践
- 分享大模型应用落地的工程经验
```

### 教程/指南
- "介绍" + 内容
- "涵盖" + 主题列表
- "适合" + 目标受众

**示例**:
```
- 介绍从零开始构建 AI 应用的完整流程
- 涵盖架构设计、数据流和部署策略
- 适合有一定基础的开发者
```

## 术语使用

### 技术术语
- 保持准确性,不随意缩写
- 首次出现使用全称,后续可缩写
- 已知缩写可直接使用(LLM, API, SDK)

**示例**:
```
✅ Large Language Model (LLM)
✅ Application Programming Interface (API)
```

### 中英文混排
- 技术术语保留英文
- 描述性内容使用中文
- 避免中英文混杂的词汇

**示例**:
```
✅ 支持 React 和 Vue 框架
❌ 支持 React 和 Vue frameworks
```

### 特殊符号
- 使用 emoji 区分状态和类型
- 使用 Markdown 语法强调
- 避免过多装饰性符号

**示例**:
```
✅ 🆕 新工具 | 🔥 热门 | 📁 仓库
❌ ✨✨✨ 新工具 ✨✨✨
```

## 描述模板

### 工具类
```
[核心功能] + [适用场景]
```
**示例**:
```
快速构建 AI 应用的开发框架
提供 API 集成和模型管理,适合生产环境
```

### 文章类
```
[主题] + [核心观点]
```
**示例**:
```
深入探讨微前端架构的落地实践
分析微前端在企业级应用中的优势和挑战
```

### 仓库类
```
[项目定位] + [核心特性]
```
**示例**:
```
Swift 应用架构库,强调组合与可测试性
提供完整的状态管理和副作用处理方案
```

## 常见错误

### 错误 1: 主观评价
```
❌ 这是一个非常优秀的工具
✅ 提供 XXX 功能,支持 YYY
```

### 错误 2: 描述冗长
```
❌ 这是一个基于 React 开发的现代化前端框架,它能够帮助开发者快速构建高质量的应用
✅ 基于 React 的现代化前端框架,快速构建高质量应用
```

### 错误 3: 夸大表述
```
❌ 支持所有功能
✅ 支持 10+ 核心功能
```

### 错误 4: 信息不足
```
❌ 一个好工具
✅ 提供 XXX 功能的 AI 工具
```
